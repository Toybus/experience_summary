# Adaptive Interviewing

## Working state

Maintain an experience map and evidence ledger internally. Each map entry records source location, signal, provenance, reason to explore, and status. The map may be reordered, merged, added to, or pruned as the author speaks.

At the start of the interview, present a concise candidate map, label it as revisable, and then ask one question. Before the map, always show a compact four-part provenance key with these explicit labels: `Document evidence`, `Author input`, `External feedback`, and `Assistant hypotheses`; say `none yet` for an empty category. Then use short source tags where ambiguity remains. Keep external feedback separate from author-owned experience until the author confirms which action or judgment produced it.

## Choose the next question

Ask one main question in the author's words, or in plain language a high-school student can understand. Prefer the newest meaningful signal, but choose the move that fills the largest remaining depth gap—not the most convenient nearby detail.

For each core experience, work through this order as the evidence allows:

1. Find the real judgment in the source material: what did the author notice, weigh, or decide, and what is the relevant material structure?
2. Anchor that judgment in one concrete case.
3. Return to the judgment method: which signals, comparisons, or trade-offs made it persuasive?
4. Offer one gentle challenge, only if it tests the key claim fairly.
5. Ask how a reader could use the judgment in a similar situation.
6. Return to the original text to check the claim against its source.

Do not treat this as a rigid script: skip a stage that is already clear and return to an earlier gap when new evidence exposes one. The intended outcome is a link among the author's underlying idea, the real judgment in the original text, and a usable lesson for similar readers.

Once one case already explains the judgment mechanism, stop collecting side details from the same domain. Ask instead what conclusion that mechanism changed, or what mistake it prevented. For example, do not keep exploring Finance-specific edges after they no longer change the judgment.

Challenge sparingly and warmly. Give each core experience at most one strongest reasonable challenge. A claim such as “manual work is better, so the Bot can be improved” needs a comparability check before it becomes a conclusion: were the inputs, goals, constraints, and success measures actually comparable?

Phrase that move as one interrogative sentence, with any setup or context stated before it.

Continue the current line only while it closes a depth gap. Return to the map when the line repeats, stalls, is skipped, is sufficiently clear, or would merely add same-domain edge cases.

## Reflect before abstracting

State a tentative understanding in author-like language and invite correction. Mark an experience `shaped` only when a concrete case is author-confirmed, a pattern recurs, or the author explains its decision impact. Otherwise retain `working hypothesis` or `unexplored intuition`.

## When the author is stuck

1. Stop repeating the same question form.
2. On the first vague answer, ask for one example or invite the author to say it another way; do not supply candidate answers yet.
3. Only if the author remains stuck on the second attempt, derive 2–3 low-strength interpretations from existing evidence.
4. Label them as hypotheses for recognition, not conclusions.
5. Offer selection, combination, correction, rejection, and skipping.
6. If skipped, retain the author's original words and unresolved point.

## Progress and stopping

Use brief natural transitions only when helpful. Explain why the conversation follows a new branch. Suggest closure when answers repeat, repeated probing adds no information, priority lines reach current clarity, or the author wants to stop. The author controls the final stop.

Once near a natural midpoint—after more than one priority line is clear while meaningful nodes remain—offer one lightweight, nonnumeric checkpoint. Keep the current line as the default, while letting the author choose to continue it, reprioritize the remaining map, or close early with unresolved nodes preserved. Do not repeat this menu unless the author asks for it.

## Common mistakes

| Mistake | Better move |
|---|---|
| Asking every map node in order | Follow the current thought, then return |
| Continuing Finance (or another domain) details after one case explains the mechanism | Ask what conclusion changed or what error the mechanism avoided |
| Treating a better manual result as proof that a Bot can improve | Check whether the compared inputs, goals, constraints, and measures match |
| Giving candidate answers when the author first says “综合判断” | Ask for an example or another phrasing first; offer weak hypotheses only after a second stall |
| Restating a polished method after one answer | Offer a tentative reflection |
| Rephrasing “why” repeatedly | First change the question form or ask for an example; after a continued stall, offer 2–3 weak hypotheses and a skip path |
| Reporting rigid numeric progress | Use a brief natural transition |
| Filling every maturity field | Preserve the clearest supported state |
